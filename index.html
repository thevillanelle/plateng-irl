The Report Card
Platform Engineering Rubric
Seven dimensions. Each scored 0–4. Maximum 28 points. Grade every platform you build.

“It’s like a report card, but for your platforms. And unlike Mr. Hall’s class, you actually get to know what you’re being graded on before the test.”

📂 Gather files for the grader — three ways
⭐ Option 1 — Ask Claude Code using gh (no clone needed)
Works if you're on your network or VPN.
Download my grading files from GitHub Enterprise without cloning the repo. Use gh api to pull from github.com. Repo: [owner/repo-name]. Save to ~/Desktop/grade-[repo-name]/. Get: Dockerfile, main.py (try api/main.py if not at root), test_main.py, .github/workflows/deploy.yml, deploy.yaml or any .github/workflows/*.yml, nginx.conf, CONTRIBUTING.md, CHANGELOG.md — skip missing files. Open the folder when done.
Option 2 — Ask Claude Code (paste this prompt)
Grab my Report Card grading files. From the current repo, copy to ~/Desktop/grade-this-repo/: main.py (or server.py), Dockerfile, test_main.py, any .yml from .github/workflows/, .github/workflows/deploy.yml, nginx.conf, CONTRIBUTING.md, CHANGELOG.md. Skip missing. Open in Finder when done.
Option 3 — Shell command (run from your repo root)
# 1. Navigate to your repo first — for example:
cd ~/projects/my-app

# 2. Copy grading files to Desktop
mkdir -p ~/Desktop/grade-this-repo
for f in main.py server.py app.py Dockerfile test_main.py test_server.py nginx.conf CONTRIBUTING.md CHANGELOG.md; do
  [ -f "$f" ] && cp "$f" ~/Desktop/grade-this-repo/
done
[ -d .github/workflows ] && cp .github/workflows/*.yml ~/Desktop/grade-this-repo/ 2>/dev/null
[ -f .github/workflows/deploy.yml ] && cp .github/workflows/deploy.yml ~/Desktop/grade-this-repo/

# 3. Open the folder — drag everything into the grader
open ~/Desktop/grade-this-repo

Share this rubric
Paste into any LLM, use in code reviews, or add to your own README. This is a sharable platform engineering standard.
1 · Application Layer
Does it do what it’s supposed to do, cleanly?
0
Not deployed or broken
1
Works but crashes on edge cases
2
Deployed and reliable, errors handled
3
Good UX, interaction states, edge cases covered
4
Full test coverage + documented API contract + accessible
2 · Data Layer
Is the data designed to survive?
0
No persistence — data lost on restart
1
Persists, but schema is ad hoc
2
Deliberate schema with the right DB for the workload
3
Versioned migrations, idempotent, safe to re-run on restart
4
Indexes on queried columns + no N+1 queries + normalized schema
3 · Reliability Layer
Can you change it without fear?
0
No tests, no CI
1
Some manual testing only
2
Automated tests on critical paths, DB_PATH configurable
3
CI on every push + user isolation test exists
4
>80% branch coverage + CD wired + rollback documented
4 · Security Layer
Who can access what?
0
Public endpoint, no authentication
1
Network protection only (firewall layer)
2
SSO authentication on the endpoint
3
AuthN + authZ + parameterized queries + no hardcoded secrets
4
All of 3 + ownership checks on all writes + OWASP top 5 addressed
5 · Operations Layer
Do you know what it’s doing right now?
0
No logging, no monitoring
1
Basic stdout logs only
2
Structured JSON logs + health check validates DB
3
SLO defined + runbook exists + alerting on error rate
4
All of 3 + OTel tracing instrumented + p99 baselined + error budget tracked
6 · Architecture
Will it hold up as it grows?
0
No separation of concerns, tightly coupled
1
Some separation but high coupling between layers
2
Clear layer separation: app, data, serving
3
Stateless app + stateful data outside container + API contracts
4
All of 3 + ADRs written + horizontal scaling tested + deprecation policy
7 · Discoverability & InnerSource
Can someone else at your company use and contribute to this?
0
No README
1
README exists but assumes context
2
README: what it does, who it's for, how to run it
3
README + CONTRIBUTING + CHANGELOG
4
All of 3 + CODEOWNERS + issue templates + SemVer + compliant PR history

Tools · Deployment Reference
Build & Deploy Reference
The standard Docker build/push pattern and a production-ready Dockerfile for FastAPI.
⚡
Build, tag & push
Replace 1.0.0 with your version number
docker build -t your-app:1.0.0 . && \
docker tag your-app:1.0.0 \
  ghcr.io/your-username/your-app:1.0.0 && \
docker push ghcr.io/your-username/your-app:1.0.0
🐳
Production Dockerfile
Multi-stage, non-root, port 8080
# --- Builder Stage ---
FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN python -m venv /app/venv
RUN /app/venv/bin/pip install --no-cache-dir -r requirements.txt
COPY . .

# --- Runtime Stage ---
FROM python:3.11-slim AS runtime
WORKDIR /app
COPY --from=builder /app /app

RUN addgroup --system appgroup && \
    adduser --system --ingroup appgroup appuser && \
    mkdir -p /data && chown -R appuser:appgroup /app /data
USER appuser

EXPOSE 8080
CMD ["/app/venv/bin/python", "-m", "uvicorn", \
     "main:app", "--host", "0.0.0.0", "--port", "8080"]
📋
Platform checklist
What most container platforms require
RequirementStatus
Non-root user (appuser)✅ In Dockerfile above
Expose port 8080✅ In Dockerfile above
Health check endpoint (/health)⚠️ Add to main.py
No credentials baked into image✅ Use secrets manager
Immutable image tag (SHA or version)⚠️ Never deploy :latest
🔄
Run locally
Test your container before deploying
docker run -p 8080:8080 \
  -e DB_PATH=/tmp/test.db \
  your-app:1.0.0
# Then open http://localhost:8080

Agent System Prompt
Copy the whole thing: it’s complete
You are a platform engineering tutor for Platform Engineering IRL — a self-directed, university-level curriculum for engineers who build and run production systems.

═══ WHO YOU ARE ═══

Rigorous, specific, direct. You explain complex concepts clearly. You tie concepts to real production code. You never give generic examples when a specific one exists. You know the student's stack: FastAPI, SQLite/PostgreSQL, Docker, GitHub Actions, Railway/Vercel/Supabase.

Modes:
- TEACH: explain deeply, with code, with context
- QUIZ: test understanding, give feedback

═══ THE COURSE ═══

22 modules + 10 electives covering:

FOUNDATION: Development environment, SSO/identity/RBAC, language stack (Python, SQL, Clojure, Swift, JS), GitHub for teams, PR hygiene & change management, developer workflow

BUILD: Systems design (APIs, caching, async, scaling, trade-offs), data layer (schema, normalization, indexing, transactions, migrations), design (Apple HIG→platform principles, tokens, spacing, interaction, accessibility), UX testing (usability testing, heuristics, A/B testing, card sorting)

HARDEN: Security (threat modeling, OWASP, secrets management, parameterized queries, per-user ownership checks), testing (TDD, test doubles, coverage, FastAPI integration testing), CI/CD (pipeline anatomy, deployment strategies, artifact management, failure recovery), deployment (GitHub Pages, Vercel, Railway, Supabase, GitHub Actions)

OPERATE: Observability (SLO/SLI/error budgets, alerting, distributed tracing, dashboards), production ops (incident management, on-call, profiling, capacity planning), managing multiple projects (triaging health, context switching, parking notes, portfolio decisions, cross-LOB communication), platform standards (code review, ADRs, API design, deprecation, SemVer)

ELECTIVES: OTel in Python, SQL Beyond Basics, Python Async, Kubernetes Basics, The Data Stack (OLAP vs OLTP, analytical SQL), InnerSource, MCP & Claude Skills

═══ KEY CONCEPTS TO DRILL ═══

- Per-user data isolation: every query filters by user_id (WHERE user_id=?)
- Parameterized queries: never string-interpolate user input into SQL
- Versioned migrations: idempotent, run on startup, never destructive
- Stateless app + stateful data outside the container
- CI gates deploys: broken tests cannot reach production
- Secrets in environment variables, never in source code
- SLO → error budget → deployment decisions
- Parking notes for context switching across multiple projects
- One concern per PR; PR description answers what AND why
- Atomic commits; conventional commit format
