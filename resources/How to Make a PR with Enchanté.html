How to make a pull request

Using Enchanté — no command line required. For contributors who are new to GitHub.

What is a pull request?

A pull request (PR) is how you propose a change to a shared codebase or document. Instead of editing directly, you submit your changes for review — someone approves them, and then they're merged in. Every change is logged, with your name, a description, and a timestamp. That audit trail is what keeps SDS compliant with GESC requirements.

Why PRs matter here

GESC compliance requires a clear record of who changed what, when, and why. A well-written PR is that record.

What Enchanté does

Enchanté can read your repo, understand the change you want to make, write the code or content, and submit the PR — in plain conversation.

1 Open Enchanté and connect to GitHub

Open Enchanté on your Mac. If you haven't installed it yet, download it from enchante.hwe.apple.com — it signs in automatically via AppleConnect.

Connect GitHub via L'Atelier
  1. Press ⇧ ⌘ L to open L'Atelier (Enchanté's tool store)
  2. Search for GitHub and click Install
  3. Follow the prompts to authenticate with your GitHub account
  4. Go to Settings → MCP and confirm GitHub shows a green status

You only need to do this once. After that, Enchanté can read and write to any GitHub repo you have access to.

2 Describe your change in plain language

Start a new conversation in Enchanté. Tell it what repo you're working in, what you want to change, and why. You don't need to know any code to do this.

Example prompts
  1. "In the sds-docs repo, update the README to add a section about our GESC screening process. Here's what it should say: [paste your text]"
  2. "In platform-runbooks, add a new file called onboarding-checklist.md with the following content: [paste content]"
  3. "Update line 14 of config.yaml in the sds-compliance repo — change the timeout value from 30 to 60."

The more specific you are, the better. Include the repo name, the file (if you know it), and a clear description of the change. Enchanté will ask clarifying questions if it needs more context.

3 Review the PR before it's submitted

Enchanté will show you exactly what it's going to change — the file name, the before and after, and the PR title and description it will use. Read it carefully before confirming.

What to check
  1. PR title — should clearly say what changed (e.g., "Add GESC screening section to README")
  2. Description — should explain why the change is being made
  3. Files changed — confirm only the intended file(s) are included
  4. Content — read the actual change and make sure it's correct

If anything looks wrong, just tell Enchanté what to fix before proceeding. Don't submit until you're satisfied.

4 Submit for review

Once you confirm, Enchanté will create a new branch, commit the change, and open the PR on GitHub. It will share a link to the PR so you can see it.

After submitting
  1. Enchanté will give you a direct link to the PR on GitHub — click it to verify
  2. Tag the right reviewer — ask your team lead who that is if you're unsure
  3. Add a comment if there's any context the reviewer needs to know

Never tag yourself as the reviewer on your own PR. PRs require a second set of eyes — that's the point.

5 Merge when approved

Once a reviewer approves the PR, you'll get a notification via GitHub (and email if you have it set up). You can then merge the PR.

Merging via Enchanté
  1. In Enchanté, say: "Merge the PR at [paste GitHub link]"
  2. Enchanté will confirm the PR is approved and merge it
  3. The branch will be deleted automatically if your repo is set up that way
Merging directly on GitHub
  1. Open the PR link in your browser
  2. Click the green Merge pull request button
  3. Click Confirm merge

What a good PR looks like

Use this as a reference when Enchanté drafts your PR title and description.

Title
Add GESC screening requirement to onboarding runbook
Description
Adds a new section to the onboarding runbook documenting the GESC sanctions screening step.

Why: New hires were missing this step during setup. GESC compliance requires it be documented and followed for all new system access requests.

What changed: onboarding-runbook.md — added "Sanctions Screening" section after Step 3.

Reviewers: @[team-lead]
⚠

A note on compliance. In SDS, PRs are part of our audit trail for GESC (Global Export & Sanctions Compliance). Every change to a production system or compliance-related document should have a PR — never a direct commit. Your PR description is a record. Write it like someone will read it later to understand what happened and why.
