How to make a pull request

Using Interlinked — for teams building AI-powered workflows that include GitHub actions.

What is a pull request?

A pull request (PR) is how you propose a change to a shared codebase or document. Instead of editing directly, you submit your changes for review — someone approves them, and then they're merged in. Every change is logged, with your name, a description, and a timestamp. That audit trail is what keeps SDS compliant with GESC requirements.

Why PRs matter here

GESC compliance requires a clear record of who changed what, when, and why. A well-written PR is that record.

What Interlinked does

Interlinked is Apple's internal developer library for building AI tools. If your team has an Interlinked-powered app or automation, it can be connected to GitHub to create and manage PRs programmatically.

1 Access Interlinked via chat.apple.com

Interlinked is accessible through chat.apple.com — Apple's internal AI chat interface. If your team has built a custom Interlinked-powered tool, your team lead will share the link or app directly.

Getting started
  1. Go to chat.apple.com in your browser (must be on Apple network or VPN)
  2. Sign in with AppleConnect — no extra setup needed
  3. Select your model and any Applets your team has configured

If your team uses a custom Interlinked-powered tool instead of chat.apple.com directly, use that. The steps for describing and reviewing changes are the same.

2 Connect to GitHub

For Interlinked to create PRs, the GitHub integration must be active. This is typically set up by your team's developer — check with them if you're unsure whether it's already configured.

If you're setting it up yourself
  1. In chat.apple.com, open the Applets panel
  2. Find the GitHub applet and enable it
  3. Authenticate with your GitHub credentials when prompted
  4. Confirm the integration is active — you should see GitHub listed as a connected tool
For developers building with Interlinked

Use the interlinked GitHub tool module to wire up repo access. Full documentation at istweb.apple.com/interlinked. Authenticate via OAuth using the user's AppleConnect identity — do not use shared service tokens for PRs that need individual attribution.

3 Describe your change in plain language

In the chat interface, describe what you want to change — include the repo name, the file, and a clear reason for the change.

Example prompts
  1. "In the sds-compliance repo, update the GESC screening doc to reflect the new Q3 requirements. Here's the updated text: [paste text]. Create a branch and open a PR."
  2. "Add me as a contributor to the CODEOWNERS file in platform-runbooks. Open a PR with a description explaining the change."
  3. "Create a new branch in sds-docs, add the attached content as a new file called screening-policy.md, and submit a PR."

Interlinked will show you what it plans to do before acting. You can ask it to adjust the branch name, PR title, description, or content before confirming.

4 Review before submitting

Before the PR is created, Interlinked will present a summary of the change. Review it carefully — the PR description becomes a permanent compliance record.

What to check
  1. PR title — specific and descriptive
  2. Description — includes what changed and why
  3. Files in scope — only what you intended
  4. Reviewer — tagged before submission

Confirm when satisfied. Interlinked will push the branch and open the PR, returning a direct GitHub link.

If your team's Interlinked tool has a compliance template for PR descriptions, use it — it ensures the right fields are filled in for GESC audit purposes.

5 Merge when approved

Once your reviewer approves the PR, merge it via Interlinked or directly on GitHub.

Merging via Interlinked
  1. In the chat, say: "Merge the PR at [paste GitHub link]"
  2. Interlinked will confirm approval status and merge
Merging directly on GitHub
  1. Open the PR link in your browser
  2. Click the green Merge pull request button
  3. Click Confirm merge

What a good PR looks like

Use this as a reference when Interlinked drafts your PR title and description.

Title
Add GESC screening requirement to onboarding runbook
Description
Adds a new section to the onboarding runbook documenting the GESC sanctions screening step.

Why: New hires were missing this step during setup. GESC compliance requires it be documented and followed for all new system access requests.

What changed: onboarding-runbook.md — added "Sanctions Screening" section after Step 3.

Reviewers: @[team-lead]
⚠

A note on compliance. In SDS, PRs are part of our audit trail for GESC (Global Export & Sanctions Compliance). Every change to a production system or compliance-related document should have a PR — never a direct commit. Your PR description is a record. Write it like someone will read it later to understand what happened and why.
