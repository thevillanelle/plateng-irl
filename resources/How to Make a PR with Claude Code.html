How to make a pull request

Using Claude Code in VS Code — no command line required. For contributors who are new to GitHub.

What is a pull request?

A pull request (PR) is how you propose a change to a shared codebase or document. Instead of editing directly, you submit your changes for review — someone approves them, and then they're merged in. Every change is logged, with your name, a description, and a timestamp. That audit trail is what keeps SDS compliant with GESC requirements.

Why PRs matter here

GESC compliance requires a clear record of who changed what, when, and why. A well-written PR is that record.

What Claude Code does

Claude Code lives inside VS Code. You describe your change in plain English and it makes the edit, creates the branch, and submits the PR — all from the chat panel.

1 Open VS Code in the right folder

Open VS Code. Before using Claude Code, make sure you have the repo folder open — not just a single file.

Opening the repo
  1. In VS Code, go to File → Open Folder
  2. Navigate to your local copy of the repo and click Open
  3. If you don't have a local copy, ask your team lead to share the repo link and help you clone it first

If the repo isn't on your computer yet, tell Claude Code: "Clone the repo at [paste GitHub URL] to my Desktop." It will handle it.

2 Open Claude Code

Claude Code is a panel inside VS Code — you talk to it like a chat window and it acts on your repo.

Opening the panel
  1. Press ⌘ ⇧ P to open the Command Palette
  2. Type Claude and select Claude Code: Open
  3. The chat panel will appear on the right side of VS Code

If you don't see Claude Code as an option, it may not be installed. Go to the Extensions panel (⌘ ⇧ X), search for Claude Code, and install it. Then complete the GenAI onboarding at genai.apple.com if you haven't already.

3 Describe your change in plain language

Type your request into the Claude Code chat panel. Be specific — include what file to change, what the change is, and why.

Example prompts
  1. "Create a new branch called docs/add-gesc-section, add a GESC screening section to README.md, then open a PR with a clear description."
  2. "Update the timeout value in config.yaml from 30 to 60, create a branch, and open a PR explaining the reason."
  3. "Add a new file called onboarding-checklist.md with the following content: [paste content]. Create a PR for it."

Claude Code will show you each action it's taking — editing the file, creating the branch, staging the commit — before it opens the PR. You can stop it at any point.

4 Review and submit

Claude Code will draft a PR title and description and show it to you before submitting. Read it carefully — this becomes part of the permanent audit record.

What to check
  1. Title — should clearly describe what changed
  2. Description — should explain why the change is being made
  3. Files changed — should only include what you intended
  4. Branch name — should be descriptive, not patch-1 or fix

If anything needs adjusting, just tell Claude Code what to change before it submits. Once you confirm, it will push the branch and open the PR on GitHub, sharing a direct link.

Add your reviewer's GitHub handle in the request: "Open a PR and request review from @[handle]." Claude Code will tag them automatically.

5 Merge when approved

Once your reviewer approves, you can merge. Claude Code can do this for you, or you can do it directly on GitHub.

Merging via Claude Code
  1. In the Claude Code panel, say: "Merge the PR at [paste GitHub link]"
  2. Claude Code will confirm it's approved and merge it
Merging directly on GitHub
  1. Open the PR link in your browser
  2. Click the green Merge pull request button
  3. Click Confirm merge

What a good PR looks like

Use this as a reference when Claude Code drafts your PR title and description.

Title
Add GESC screening requirement to onboarding runbook
Description
Adds a new section to the onboarding runbook documenting the GESC sanctions screening step.

Why: New hires were missing this step during setup. GESC compliance requires it be documented and followed for all new system access requests.

What changed: onboarding-runbook.md — added "Sanctions Screening" section after Step 3.

Reviewers: @[team-lead]
⚠

A note on compliance. In SDS, PRs are part of our audit trail for GESC (Global Export & Sanctions Compliance). Every change to a production system or compliance-related document should have a PR — never a direct commit. Your PR description is a record. Write it like someone will read it later to understand what happened and why.
